3) FoodService - главный бизнес сервис. Точка входа для клиента.
Шаги:
API имеет хендлер /login, внутри делает HTTP-вызов к auth-сервису.
API проверяет JWT локально (публичный ключ получает при старте от auth-сервиса или из файла).
API — фактически и есть Gateway, только встроенный в код.

Выделяем auth-логику в middleware (подготовка к Gateway)
Оставьте в API поддержку двух режимов через переменную окружения

Чтобы Gateway был легко добавить позже, ваш API должен:

✅ Хорошо:
Принимать user_id в заголовке (например, X-User-Id).
Иметь конфигурацию "trusted gateway subnet" (192.168.0.0/16) — заголовки приходят только от Gateway, а не от клиента.
Быть stateless (не хранить сессии, токены, ничего локального).

❌ Не доверяйте X-User-Id от клиента. Только если запрос пришел с доверенного IP (ваш Gateway). Иначе любой сможет подставить чужой ID.
